在计算机科学中，future，promise，delay和deferred指的是用于在某些并发编程语言中同步程序执行的结构。他们描述了一个对象，该对象充当最初未知的结果的代理，通常是因为其值的计算尚未完成。

术语承诺是由在1976年提出的丹尼尔P.弗里德曼和戴维·怀斯，[1] 和彼得希巴德把它称为最终的。[2]亨利·贝克和卡尔·休伊特（Henry Baker）和卡尔·休伊特（Carl Hewitt）在1977年的一篇论文中介绍了 某种类似的概念未来。[3]

术语Future，promise，delay和deferred经常可以互换使用，尽管future和promise在用法上的一些差异在下面进行了处理。具体来说，当区分用法时，future是变量的只读占位符视图，promise是可写的单个分配设置未来价值的容器。值得注意的是，可以定义一个未来而无需指定哪个特定的诺言将设置其价值，并且不同的可能的诺言可以设置给定未来的价值，尽管对于一个给定的未来只能进行一次。在其他情况下，将来和承诺是一起创建并彼此关联的：将来是价值，承诺是设置值的函数-本质上是异步函数（承诺）的返回值（未来）。设定未来的价值也称为解决，实现或约束它。

应用程序[编辑]

期货和承诺起源于函数式编程和相关的范式（例如逻辑编程），以将值（未来）与计算方式（承诺）分离开来，从而允许更灵活地完成计算，尤其是通过并行处理。后来，它发现可用于分布式计算中，以减少通信往返的延迟。后来，它通过允许以直接样式而不是连续传递样式编写异步程序而获得了更多的使用。

隐式与显式[编辑]

期货的使用可以是隐式的（其中使用的任意未来的自动获取其值，就好像它是一个普通的参考）或显式（用户必须调用一个函数来获得的值，如get方法java.util.concurrent.Future中的Java）。获得明确未来的价值可以称为刺激或强迫。可以将显式期货实现为库，而将隐式期货通常实现为语言的一部分。

最初的Baker和Hewitt论文描述了隐式的未来，计算的参与者模型和像Smalltalk这样的纯面向对象的编程语言自然支持这种隐式的未来。Friedman and Wise的论文仅描述了显性期货，这可能反映了在股票硬件上有效实施隐性期货的困难。困难在于，股票硬件无法处理诸如整数之类的原始数据类型的期货。例如，添加指令不知道如何处理。使用纯actor或对象语言，可以通过发送消息来解决此问题，该消息要求将来添加3 + future factorial(100000)future factorial(100000)+[3]3本身并返回结果。请注意，消息传递方法不管何时factorial(100000)完成计算都有效，并且不需要任何刺激/强制。

承诺流水线[编辑]

利用期货，可以大大减少等待时间的分布式系统。例如，期货使诺流水线，[4] [5]如在语言实现ë和焦耳，这也被称为呼叫流[6]中的语言阿格斯。

考虑一个涉及常规远程过程调用的表达式，例如： 

 t3：=（xa（））.c（yb（））

可以扩展到 

 t1：= xa（）;
 t2：= yb（）;
 t3：= t1.c（t2）;

每个语句都需要发送一条消息并接收一条回复，然后才能继续执行下一条语句。假设，例如，即x，y，t1，和t2全部位于相同的远程计算机上。在这种情况下，必须先进行两次到该计算机的完整网络往返，然后才能开始执行第三条语句。然后，第三条语句将导致到同一远程计算机的另一次往返。

使用期货，上面的表达式可以写成 

 t3：=（x <-a（））<-c（y <-b（））

可以扩展到 

 t1：= x <-a（）;
 t2：= y <-b（）;
 t3：= t1 <-c（t2）;

此处使用的语法是语言E的语法，其中Ex <- a()表示将消息a()异步发送到x。这三个变量将立即为其结果分配期货，然后执行继续执行后续语句。以后尝试解决的值t3可能会导致延迟。但是，流水线可以减少所需的往返次数。如果，如在现有例子中，x，y，t1，和t2全部位于相同的远程计算机上，以流水线的实现可以计算t3与一个往返，而不是三个。因为所有这三个消息都发往同一远程计算机上的对象，所以只需要发送一个请求，而只需要接收一个包含结果的响应。发送t1 <- c(t2)将不会阻止即使t1并t2分别在不同的机器到彼此，或者x或y。

应将承诺管道与并行异步消息传递区分开来。在支持并行消息传递的系统，而是不流水线，消息发送x <- a()和y <- b()在上面的例子中可以并行进行，但是发送t1 <- c(t2)将必须等待，直到两个t1和t2已收到，即使当x，y，t1，和t2在相同的远程机器。在涉及许多消息的更复杂的情况下，流水线的相对延迟优势变得更大。

承诺流水线也不应与参与者系统中的流水线消息处理相混淆，在这种情况下，参与者可以在完成对当前消息的处理之前指定并开始执行下一条消息的行为。

只读视图[编辑]

在某些编程语言（例如Oz，E和AmbientTalk）中，可能会获得Future的只读视图，该视图允许在解析后读取其值，但不允许解析它的值：

  • 在Oz中，!!运算符用于获取只读视图。
  • 在E和AmbientTalk中，前途由一对称为Promise / Resolver对的值表示。Promise表示只读视图，需要使用解析器来设置未来的价值。
  • 在C ++ 11中， astd::future提供了只读视图。该值是通过使用一个直接设置std::promise，或设置为利用函数调用的结果std::packaged_task或std::async。
  • 在1.5版的Dojo Toolkit的Deferred API中，仅面向消费者的诺言对象表示只读视图。[7]
  • 在Alice ML中，Future提供了只读视图，而Promise包含了Future和解决该Future的能力[8] [9]
  • 在.NET Framework 4.0中， System.Threading.Tasks.Task<T>代表一个只读视图。可以通过解析值System.Threading.Tasks.TaskCompletionSource<T>。

对于只读视图的支持是一致的最小权限原则，因为它能够设定被限制到价值的能力科目是需要进行设置。在还支持流水线的系统中，异步消息的发送者（带有结果）将接收结果的只读承诺，而消息的目标将接收解析器。

特定于线程的期货[编辑]

某些语言（例如Alice ML）定义了与特定线程关联的期货，该特定线程计算了期货的价值。[9]这种计算既可以在创建未来时急切地开始，也可以在第一次需要它的值时懒惰地开始。从延迟的计算的意义上讲，懒惰的未来类似于笨拙的未来。

Alice ML还支持可由任何线程解决的期货，并称其为promise。[8]诺言的这种使用不同于如上所述在E中的使用。在Alice中，Promise不是只读视图，并且不支持Promise流水线。取而代之的是，对于期货，包括与承诺相关的期货，自然会发生流水线操作。

阻塞与非阻塞语义[编辑]

如果某个future的值是异步访问的，例如通过向其发送消息，或使用诸如whenE中的构造显式地等待它，那么就可以毫不费力地延迟到解决future之前，才可以对消息进行处理。收到或等待完成。这是在纯异步系统（例如纯参与者语言）中要考虑的唯一情况。

但是，在某些系统中，也可能尝试立即或同步访问期货的价值。然后有一个设计选择：

  • 该访问可能会阻塞当前线程或进程，直到将来解决为止（可能存在超时）。这是语言Oz中数据流变量的语义。
  • 尝试进行的同步访问可能总是发出错误信号，例如抛出异常。这是E.中远程承诺的语义。[10]
  • 如果将来已经解决，则访问可能会成功，但是如果尚未解决，则可能会发出错误信号。这将带来引入不确定性和潜在竞争条件的缺点，并且似乎是不常见的设计选择。

作为第一种可能性的示例，在C ++ 11中，需要一个future值的线程可以阻塞，直到通过调用wait()或get()成员函数将其可用为止。您也可以使用wait_for()或wait_until()成员函数在等待时指定超时，以避免不确定的阻塞。如果将来发生在调用之后，std::async则阻塞等待（无超时）可能导致函数的同步调用，以在等待线程上计算结果。

一个I-VAR（如在语言标识）是阻塞语义如上定义的未来。一个I-结构是一个数据结构含有I-瓦尔。可以多次设置不同值的相关同步构造称为M-var。M-var支持原子操作以获取或放入当前值，其中获取值还将M-var设置回其初始空状态。[12]

未来不同形式的表现力之间的关系[编辑]

通过在创建期货的同时创建一个线程来计算价值，可以在非特定于线程的期货中直接实现渴望于线程的期货。在这种情况下，希望将只读视图返回给客户端，以便只有新创建的线程才能解决此问题。

要用非特定线程的期货来实现隐式的特定于特定线程的特定线程（例如，由Alice ML提供），需要一种机制来确定何时首先需要该期货的值（例如，WaitNeededOz中的构造[13]。 ]）。如果所有值都是对象，则实现透明转发对象的能力就足够了，因为发送到转发器的第一条消息表明需要将来的值。

假设系统支持消息传递，则可以通过使解析线程将消息发送到将来的自己的线程来在特定于线程的将来中实现非特定于线程的未来。但是，这可以看作是不必要的复杂性。在基于线程的编程语言中，最具表现力的方法似乎是提供非特定于线程的期货，只读视图，WaitNeeded结构或对透明转发的支持的混合体。

评估策略[编辑]

更多信息：将来致电

期货的评估策略（可以称为按期货赎回）是不确定的：期货的价值将在创建期货和使用其价值之间的某个时间进行评估，但无法确定确切的时间。可以事先更改，并且可以逐次更改。计算可以在创建未来（急切评估）时立即开始，或者仅在实际需要该值时才开始（懒惰评估），并且可以中途暂停或一次执行。一旦指定了Future的值，就不会在以后的访问中重新计算它。这就像按需调用中使用的备忘录。

一个懒惰的未来是一个未来的确定性有懒评价的语义：首先需要的值时，未来的价值的计算开始，如由需要呼叫。惰性期货在语言中很有用，默认情况下评估策略不是惰性的。例如，在C ++ 11中，可以通过将std::launch::deferred启动策略传递给std::async以及创建计算值的函数来创建这样的延迟期货。

历史[编辑]

在未来的和/或承诺构建物首先在编程语言，如实施MultiLisp和法1。在并发 逻辑编程语言中使用逻辑变量进行通信与期货非常相似。这些以Freeze和IC Prolog在Prolog中开始，并通过关系语言，并发Prolog，受保护的Horn子句（GHC），Parlog，Strand，Vulcan，Janus，Oz-Mozart，Flow Java成为真正的并发原语和Alice ML。来自数据流编程语言的单分配I-var，起源于Id，并包含在Reppy的Concurrent ML中，非常类似于并发逻辑变量。

承诺流水线技术（使用期货来克服延迟）是由Barbara Liskov和Liuba Shrira于1988年发明的[6]，并由Mark S. Miller，Dean Tribble和Rob Jellinghaus分别在1989年的Xanadu项目中发明。[14]

“诺言”一词是由Liskov和Shrira创造的，尽管他们通过名称call-stream来提及流水线机制，但现在很少使用。

Liskov和Shrira的论文中描述的设计以及Xanadu中的promise管道实现都具有promise值不是一流的限制：调用或send的参数，或者返回的值不能直接是promise （因此，前面给出的承诺流水线示例将一个发送结果的承诺用作另一个发送的参数，在调用流设计或Xanadu实现中无法直接表达出来）。似乎没有在任何公开发行的Argus [15]（ Liskov和Shrira论文中使用的编程语言）中实现promise和call-streams 。阿格斯的开发在1988年左右停止。[16]Xanadu承诺管道的实现仅在Udanax Gold [17]的源代码于1999年发布时才公开可用，并且从未在任何已发布的文档中进行解释。[18] Joule和E中的后续实现完全支持一流的Promise和解析器。

几种早期的参与者语言，包括《法案》系列[19] [20]，都支持并行消息传递和流水线消息处理，但不承诺流水线。（尽管从技术上讲，可以在前两个中实现这些功能中的最后一个，但没有证据表明该法案的语言是这样做的。）

2000年之后，由于使用了用户界面的响应能力以及使用Web开发（由于消息传递的请求-响应模型），对期货和承诺的兴趣有了很大的复兴。现在，几种主流语言都对期货和承诺提供了语言支持，其中最著名的FutureTask是Java 5（2004年发布）[21]和.NET 4.5中的async / await构造（2010年发布，2012年发布）[22] [23]广泛流行。受F＃[24]的异步工作流启发，该工作流可以追溯到2007年。[25]随后，它被其他语言采用，尤其是Dart（2014），[26] Python（2015），[27] Hack（HHVM）和ECMAScript 7（JavaScript），Scala和C ++的草案。

协程[编辑]

可以在协程[27]或生成器[99]中实现期货，从而得出相同的评估策略（例如，协作式多任务处理或惰性评估）。

频道[编辑]

主条目：频道（编程）

期货可以很容易地在渠道中实现：期货是一个单一元素的渠道，而诺言则是发送到该渠道，实现未来的过程。[100] [101]这样可以在支持诸如CSP和Go之类的渠道的同时编程语言中实现期货。由此产生的期货是明确的，因为必须通过从通道读取而不是仅通过评估来访问它们。
